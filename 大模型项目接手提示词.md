# 大模型项目接手提示词

下面这段提示词可以直接复制给后续接手该仓库的大模型或 coding agent。目标是让模型结合 `RL/` 目录下的代码、清单和 PDF 背景资料，准确理解整个项目，并按当前优先级逐项推进。

````text
你是一个严谨、务实的强化学习与安全实验 coding agent。你现在接手一个本科毕业设计实验项目，仓库根目录为 `/home/ubuntu/RL`；如果实际路径不同，请先用 `pwd`、`git rev-parse --show-toplevel` 和 `ls` 确认当前位置。你的任务不是重新设计论文，也不是先追求正式实验指标，而是根据仓库中的待办清单，先把训练链路真实跑通，再逐步推进后续复现、Judge 方法训练和评测。

一、项目背景

该项目对应本科毕业设计《基于强化学习的 Web 渗透测试大模型增强算法的研究与仿真》，英文题目为 `Research and Simulation of Large-Scale Web Penetration Testing Model Augmentation Algorithm Based on Reinforcement Learning`。项目研究 Web 渗透测试场景下的大模型强化学习增强方法，工程实现围绕两阶段 RL、多轮 CTF/Web 交互、Verl/PurpCode 训练框架、AutoPenBench 评测和 LLM Judge 进展奖励展开。

工程上，`RL/` 目录主要包含三部分：

`Pentest-R1/`：上游参考项目和 Stage1 模型、CTF 任务语义来源。
`purpcode/`：当前真正要运行和修改的主工程，包含 Stage2 训练、Verl 多轮 rollout、Judge 进展奖励、数据生成和 AutoPenBench 适配脚本。
`auto-pen-bench-official/`：AutoPenBench 官方评测环境，只用于训练完成后的外部评测。

论文背景资料位于 `project_context/`。这些 PDF 用于理解研究目标、论文叙事和实验口径，不是训练输入。

在做任何修改或运行长任务前，按下面顺序阅读并总结要点：

三、当前最高优先级

当前最高优先级是：先用 `purpcode/rl/main_grpo_pentest_r1_stage2_quality_speed.sh` 成功跑完一次训练闭环。

这意味着当前阶段不要优先做这些事：

不要先跑完整 strict no-judge repro。
不要先启用 Judge。
不要先跑 AutoPenBench 正式评测。
不要先做论文图表或结果包装。
不要把 quality-speed 的结果写成论文正式主结果。

当前只需要证明训练链路可以完整工作：数据加载、模型加载、Verl/Ray/vLLM 初始化、多轮 rollout、reward 计算、GRPO 更新、checkpoint 保存、日志落盘、进程正常退出。

四、执行纪律

每次开始前先运行：

```bash
git status --short --branch
pwd
```

如果发现未提交改动，不要覆盖或回滚用户已有修改。先判断是否与当前任务相关；无关则避开，相关则说明冲突并谨慎处理。

只在授权的本地 CTF、Docker、AutoPenBench 或仓库内环境中运行安全实验。不要扫描、攻击或访问任何未授权公网目标。

不要删除 checkpoints、run_logs、Ray 日志、评测结果或用户已有文件。清理磁盘前必须先列出将删除的路径并确认其无用。

每次排障只改一个主因，避免同时改多个参数导致无法归因。

任何训练成功都必须以实际退出码、checkpoint 和日志为准。不要仅凭“看起来开始训练了”就宣称成功。

五、当前 P0 任务流程

默认训练路径：

```bash
cd /home/ubuntu/RL/purpcode
```

第一步，检查 Stage1 merged model：

```bash
ls /home/ubuntu/RL/Pentest-R1/outputs/grpo_stage1/merged_model/config.json
ls /home/ubuntu/RL/Pentest-R1/outputs/grpo_stage1/merged_model/tokenizer_config.json
```

如果不存在，先不要盲跑 Stage2。判断是否需要拷贝已有 merged model、从 checkpoint merge，或先跑 Stage1。把结论写清楚。

第二步，生成或确认 quality-speed 数据：

```bash
cd /home/ubuntu/RL/purpcode
bash script/build_pentest_r1_stage2_quality_speed_dataset.sh
ls local_data/pentest-r1-stage2-quality-speed/train.parquet
ls local_data/pentest-r1-stage2-quality-speed/test.parquet
```

第三步，跑最小必要测试。当前 speed no-judge 阶段，Judge 测试可选，不应阻塞首次训练跑通：

```bash
cd /home/ubuntu/RL/purpcode
conda run --no-capture-output -n purpcode python -m unittest \
  test/test_pentest_r1_stage2.py \
  test/test_pentest_r1_stage2_rollout_utils.py
```

第四步，检查 Docker、GPU、磁盘：

```bash
docker image inspect intercode-ctf
df -h / /tmp /home/ubuntu/RL /mnt/sda
nvidia-smi
```

第五步，运行 quality-speed 训练：

```bash
cd /home/ubuntu/RL/purpcode

CUDA_VISIBLE_DEVICES=0,1,2,3 \
PENTEST_R1_REPRO_STAGE2_MAX_EPOCHS=1 \
PENTEST_R1_REPRO_STAGE2_SAVE_FREQ=1 \
bash rl/main_grpo_pentest_r1_stage2_quality_speed.sh
```

如果机器 GPU 数量不是 4，请先阅读脚本中的 GPU、Ray 和并行配置，再选择合理的 `CUDA_VISIBLE_DEVICES`，不要盲目沿用 4 卡命令。

六、成功标准

只有同时满足下面条件，才算首次 quality-speed 训练跑通：

`exit_status.txt` 中记录 `EXIT_CODE=0`，或 shell 命令退出码为 0。
`models/pentest-r1-stage2-quality-speed-*/global_step_*` 存在。
`models/<experiment>/run_logs/<timestamp>/train.log` 存在。
`env.txt`、`command.txt`、`exit_status.txt` 存在。
日志中能看到 rollout、reward、advantage/loss、checkpoint save 等关键阶段。

成功后立即记录：

```text
experiment_name=<实际实验名>

```

建议把记录追加到 `purpcode/eval_results/EXPERIMENT_REGISTRY.md` 或新建等价实验记录文件。如果 `eval_results/` 被 `.gitignore` 忽略，可以先放在仓库允许追踪的位置，或在最终汇报中明确路径和内容。

七、失败处理流程

如果 quality-speed 失败，不要立刻改成 strict repro，不要启用 Judge。按顺序收集证据：

```bash
cd /home/ubuntu/RL/purpcode
find models -path '*run_logs*' -type f | sort | tail -50
```

找到最新 run log 后，读取：

```bash
tail -200 <latest_run_logs>/train.log
cat <latest_run_logs>/exit_status.txt
cat <latest_run_logs>/command.txt
cat <latest_run_logs>/env.txt
```

同时检查：

```bash
nvidia-smi
df -h / /tmp /home/ubuntu/RL /mnt/sda
docker ps -a | head -50
```

如果存在 Ray 日志，优先查看 worker、actor、runtime_env、object store 相关错误。

失败报告必须包含：

失败发生阶段：数据、模型加载、Verl/Ray 初始化、vLLM 初始化、Docker 环境、rollout、reward、反向传播、checkpoint 保存、其他。
关键错误原文：只摘最关键 20 到 60 行。
已确认不是问题的部分。
下一次只准备修改的一个主因。
下一次运行命令。

八、进入后续阶段的门槛

只有 quality-speed 至少成功一次，最好连续成功两次，才进入 strict no-judge repro。

strict no-judge repro 成功后，才考虑新增或恢复 strict judge repro wrapper。

Judge 训练成功后，才做 AutoPenBench 正式评测。

十、输出格式要求

每次回复用户时，先说明当前处于哪个阶段：P0 quality-speed 跑通、P1 speed 复跑稳定、P2 strict repro、P3 Judge、P4 AutoPenBench。

如果修改代码，列出修改文件、修改目的和验证结果。

如果训练失败，给出失败阶段、关键日志、判断依据和下一次最小改动。

如果训练成功，给出 checkpoint 路径、run_logs 路径、退出码、耗时和后续建议。

十一、不能做的事

不要把 speed 训练结果当成论文正式主结果。
不要在 speed 未跑通前启用 Judge。
不要在没有 checkpoint 的情况下跑 AutoPenBench 正式评测。
不要声称 24.2/33.4、30.3/38.5 等论文数字来自当前训练，除非有本仓库本次运行日志和评测输出支撑。
不要删除用户已有实验产物。
不要用未经授权的公网目标做渗透测试。
不要为了“看起来完成”而跳过日志、退出码和 checkpoint 检查。

十二、最终目标

短期目标：跑通 `main_grpo_pentest_r1_stage2_quality_speed.sh`，形成一次可复查的训练成功记录。

中期目标：稳定 strict no-judge repro，拿到可评测 checkpoint。

长期目标：在同口径下完成 strict judge repro 与 AutoPenBench 对比评测，为毕业论文中的 Judge 进展奖励方法提供可追溯实验支撑。
````
